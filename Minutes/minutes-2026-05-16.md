## Qt5 Support and Deprecation

The discussion began with a clarification on the project's stance regarding Qt5 support. While officially unsupported, the project has been accepting and reviewing Pull Requests (PRs) aimed at maintaining Q5 compatibility. This approach has led to some confusion, with a sentiment that it might suggest continued support for an obsolete version.

*   A proposal was made to "wholly drop" Qt5 support and cease accepting Qt5 PRs entirely, with a suggestion that removing Qt5 from the source code would clarify the project's stance.
*   However, an opposing view suggested that accepting PRs for Qt5 doesn't require significant developer resources and shows that some community members still desire its support.
*   It was noted that recent Qt5 PRs have primarily been from individuals trying to fix perceived regressions, rather than those actively using Qt5 and encountering issues.
*   A compromise was reached: a deprecation period where CMake would issue warnings for Qt5 usage, followed by its removal.
*   @chennes will create a PR for discussion

## AI Policy Development

A significant portion of the meeting was dedicated to discussing another [proposed AI policy](https://cloud.freecad.org/s/g4nRWHNLmjnL4XY) based on the research and consensus-building of @pieterhijma. The core of the policy aims to be "[strict permissive](https://oss-ai-policies.netlify.app/)," limiting AI-generated code to assistance only, prohibiting raw AI code due to a variety of concerns outlined in the write-up.

*   The policy acknowledges ethical concerns surrounding AI and promotes awareness of alternatives, such as the [Apertus][https://apertvs.ai/] open-source large language model.
*   There was a debate about whether explicitly acknowledging AI's ethical problems while still allowing its use is contradictory.
*   One perspective argued that the project should raise awareness of AI's issues and alternatives without imposing values or forbidding usage, emphasizing that policy should reflect community consensus.
*   Another viewpoint suggested that the project already has rules and restrictions, and adding AI ethical considerations is consistent with existing practices.
*   The idea of supporting open-source AI initiatives, like those from Infomaniak and the Swiss AI project, was discussed.
*   A suggestion was made to explore collaboration with these open source AI developers, potentially for fine-tuning AI models specifically for FreeCAD code.
*   The possibility of a PR being created for the AI policy was raised, with an aim to merge it quickly to establish a baseline for enforcement.
*   @pieterhijma will create a PR, and encourages rapid adoption since it represents a clear improvement over the existing policy, even if it is not perfect. It was emphasized that work can continue on it and that policies always change over time.

## Dependency Policy and Ubuntu LTS Support

The discussion then shifted to the dependency policy, focusing on which versions of Ubuntu LTS (Long Term Support) the project should support.

*   Historically, the project has pegged its supported libraries to the oldest Ubuntu LTS available, though this has not always been strictly followed, and was done for expediency and decision-making convenience rather than a specific desire to support that particular distribution.
*   A key issue is the conflict between supporting older LTS versions and the desire to use newer libraries, particularly for Qt/PySide and Coin.
*   The idea of dropping support for older Ubuntu LTS versions and relying on the latest or second latest LTS was proposed, along with the use of tools like Pixi for newer dependency access.
*   Concerns were raised about pinning to very specific versions, making builds fragile and difficult to replicate.
*   The importance of core libraries like Open Cascade Technology (OCCT) and Coin3D was highlighted, with a consensus that the project needs more control over these specific dependencies, potentially through vendoring or by requiring specific versions regardless of the Ubuntu LTS release.
*   @hyarion will adapt FEP-0005 with the results of this discussion

## Maintenance of Non-Core Repositories

The conversation turned to the management of the project's numerous repositories beyond the core FreeCAD repository.

*   Currently, there are 73 repositories with varying levels of activity, and no official policy for managing those with low activity or those in cold storage.
*   The core maintainers do not always possess the domain expertise to review PRs for specialized repositories like CAM machines or materials.
*   A proposal was made to create a project-wide board to improve visibility of PRs and repositories, potentially with automated actions to identify stale items and solicit reviews.
*   This approach aims to encourage community members to review and potentially become maintainers of these auxiliary repositories.
*   @kadet1090 will work with the Maintainer and Admin team to set up this project board

## Release Process and FEP Amendments

Finally, the meeting addressed two process-related items: amendments to the FEP (FreeCAD Enhancement Proposal) process and updates to the release process.

*   **FEP Process Amendment:** It was proposed to formalize the requirement that FEPs must be presented at a developer meeting to move from a draft to a proposal stage. This aims to increase visibility and encourage discussion. After a two-week feedback period, the amendment will be merged if no significant objections arise. @kadet1090 has made a PR to ammend the FEP, and will merge it in two weeks if no objections arise.
*   **Release Process:** The plan is to move away from the concept of an "LTS" release and instead focus on a "quality-focused" release with a feature freeze. This release will not have a special name but will be the first of the year. @kadet1090 has made a FEP describing this process, and will merge it in two weeks if no objections arise. The date set for the 26.3 branch to be created is 30 September 2026 -- this is the next anticipated release of FreeCAD.
*   The versioning schema will be updated to a "CalVer" format (e.g., 26.1, 27.2), reflecting that the project doesn't strictly adhere to semantic versioning for the software itself, but rather for its API.
*   A note was made that the API version is not currently exposed but will be addressed in future planning. The new release process is intended to be implemented starting in September.
