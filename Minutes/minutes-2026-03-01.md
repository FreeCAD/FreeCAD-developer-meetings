# Supplemental Materials Add-on

Dave presented a proposal for a supplemental materials add-on to address issues with the current FreeCAD material repository being baked into the core. This leads to significant delays between material commitment and user availability, potentially a year for new materials. The proposed solution is an external materials module, similar to an add-on, which would make materials immediately available through the add-on manager. Dave has already developed an external materials module for wood types and suggests this approach for generic materials as well. He proposed that materials added after version 1.1 should go into this add-on instead of the core. The add-on would be under the FreeCAD name, and if a file relies on a material not installed, it would default to the default material. The discussion also touched upon the potential for this add-on to grow large, but Dave suggested that if it becomes unmanageable, it could be split into multiple smaller modules. Phonedroid mentioned that a large material set in a specialized workbench slowed down FreeCAD for him, and Dave acknowledged that optimizations are being worked on for version 1.2. Yorik suggested creating a new repository on FreeCAD.org for this add-on.

# Implementation of Qt Testing UI Testing Environments

Dave introduced a tool called QtTesting, originally developed for ParaView, which can be used for UI testing in FreeCAD. This tool integrates with the Qt framework event loops, allowing for recording and playback of user interactions. It has two phases: record, which captures events and passes them through a translator to extract relevant information, and play, which replays these events. The recording can be in command mode (simulating user input) or verification mode (checking field values). Dave demonstrated a proof-of-concept PR, showing how to record tests by specifying an XML file and highlighting elements to check. He noted quirks such as the need to specify file extensions for recorders and potential issues with dynamic widget names. The tool allows for custom translators and players to handle specific widget behaviors. Dave believes this has significant promise for FreeCAD's testing, and it could be a good candidate for Google's Summer of Code. Benjamin found it very useful for UI testing and asked about integration with testing frameworks like `pytest`. Dave confirmed the Python interface allows for integration with Python unit testing, though he cautioned about dynamic names requiring careful setup of known preconditions. The discussion also explored the possibility of integrating parts of this into the core and UI buttons as an add-on, or having it hidden behind settings. Jackson inquired about GUI testing on Mac OS and Windows in CI, and Louis, a ParaView developer, confirmed that a graphical environment is needed, suggesting the use of XVFB on Linux for off-screen rendering and image comparison. Chris and Jackson discussed the possibility of self-hosted runners for Mac OS and Windows to enable GUI tests. Dave proposed creating an FEP for integrating QtTesting and mentioned that he has a PR available for testing.

# Steam Distribution of FreeCAD

Jackson proposed distributing FreeCAD on Steam, which would automate the process for users to get and subscribe to updates, similar to how Blender and other tools are distributed. This would also allow users to choose different channels, such as stable or weekly updates. The initial fee for adding FreeCAD to Steam was discussed as $100. Jackson began the process but realized it should be set up under the FPA (FreeCAD Project Association) using their nonprofit articles of incorporation, rather than a personal account. He requested a meeting with Yorik to set up the account. Phonedroid mentioned that as a publisher/developer on Steam, one can customize pages with assets and preview videos. He also noted that the Blender team automates their Steam distribution through their CI system, making it a largely hands-off process for managing updates and announcements.

# Remove .lex.c and .tab.c from the Repository

Benjamin proposed removing generated `..lex.c` and `.tab.c` files from the repository. He cited a recent pull request with a large number of lines in these generated files, arguing that since FreeCAD now uses Pixi (with flex and bison), these files are no longer necessary to be checked into source control. Removing them would simplify pull request reviews and reduce potential attack vectors, as users could not inject malicious code into these files. Dave raised concerns about how this would work on Windows platforms, particularly if not using Pixi for builds, and suggested consulting Chris. Benjamin countered that removing them would make it easier for others to modify the original files, as they wouldn't need to build the generated files themselves. Jackson confirmed that CI builds use a Windows LibPack, but release builds are done with Pixie, suggesting a potential transition to full-time Pixie. Phonedroid inquired if there were other generated files that should also be removed. Jackson expressed support for removing auto-generated files but suggested a CI check as a workaround if LibPack compatibility issues arise, standardizing on a specific version of Bison and Flex. Yorik agreed with the idea but emphasized the need for testing in various scenarios.
